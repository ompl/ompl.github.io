1 # OMPL Style Guide {#styleGuide}
2 
3 This document provides a brief overview of OMPL coding standards. It is meant to assist developers who are contributing code to OMPL.
4 
5 \note An automatic code formatter is available to automate enforcing the style guide, see below.
6 
7 ## Spacing
8 
9 - Each source block should be indented by 4 space characters. No tab characters should appear in source code.
10 - Each enclosing brace should reside on its own line, but braces can be omitted if the enclosed block consists of exactly one line.
11 - Each binary operator ('+', '-', '&', '=', '==', etc.) should be surrounded by spaces.
12 - The '*' and '&' in pointers and references in function/method arguments or variable declarations should be preceded by a space. They should not be followed by a space.
13 
14 The following is an example of desirable code spacing.
15 
16 ~~~{.cpp}
17 int foo(int &n)
18 {
19  for (unsigned int i = 0; i < n; ++i)
20  {
21  if (i % 2 == 0)
22  std::cout << i << " ";
23  }
24 }
25 ~~~
26 
27 ## Filenames & Macros
28 
29 - Source filenames should begin each word with a capital letter, and there should be no underscores between words.
30 - Header files should use the .h extension, and implementation files should use the .cpp extension.
31 - Each implementation file should reside in a src/ directory, immediately below the directory containing the corresponding header file.
32 - Include guards in header files should be of the form OMPL_PATH_FILENAME_, where the words in the filename are separated by underscores.
33 
34 For example, the header and implementation files in which ompl::RNG is defined are ompl/util/RandomNumbers.h and ompl/util/src/RandomNumbers.cpp, respectively. The header file begins with the include guard
35 
36 ~~~{.cpp}
37 #ifndef OMPL_UTIL_RANDOM_NUMBERS_
38 ~~~
39 
40 ## Functions & Classes
41 
42 - Function names should begin with a lower-case letter and should begin each subsequent word with a capital letter, and there should be no underscores between words (e.g., ompl::base::StateSpace::setName()).
43 - Functions accepting no parameters should have void in place of a parameter list.
44 - Class names should begin each word with a capital letter, again with no underscores (e.g., ompl::base::StateSpace).
45 - Scoping directives (_public_, _protected_, _private_) should be at the same level of indentation as the class declaration.
46 - Class member variable names should begin with a lower-case letter and should begin each subsequent word with a capital letter. They should end with a single underscore (e.g., ompl::base::StateSpace::longestValidSegment_).
47 - Names of constants and static variables should be in all capital letters, with an underscore between each word (e.g., ompl::base::StateSpace::DEFAULT_PROJECTION_NAME).
48 
49 For example, consider the following source code, which follows the above guidelines.
50 
51 ~~~{.cpp}
52 class SampleObject
53 {
54 public:
55  SampleObject(void) : objectTag_(NUM_INSTANCES++)
56  {
57  }
58 
59  int getObjectTag(void) const
60  {
61  return objectTag_;
62  }
63 
64  static int NUM_INSTANCES;
65 
66 private:
67  const int objectTag_;
68 };
69 int SampleObject::NUM_INSTANCES = 0;
70 ~~~

72 ## Other Coding Guidelines
73 
74 - Compiler specific features should never be used (avoid use of \#ifdef).
75 - Code must compile without warnings.
76 - If a class member function can be marked _const_, then it should be marked _const_.
77 - Member functions marked as _const_ must be thread safe. If a function is intended to be thread safe, it should be marked as _const_ (one can use the _mutable_ keyword if needed).
78 - All classes, methods and member variables must be documented (Doxygen style). High-level documentation can be written either in [MarkDown format](http://www.stack.nl/~dimitri/doxygen/markdown.html) or in Doxygen format.
79 - Forward declaration of types should be done with the OMPL_CLASS_FORWARD() macro when shared pointers to the type are also needed: OMPL_CLASS_FORWARD() will define a shared pointer to the type using the _Ptr_ suffix (e.g., ompl::base::StateSpacePtr).
80 - When passing objects as arguments to functions, the following convention should be used:
81  - if possible, pass a _const reference_ to the type; this means that a pointer to the object passed in will not be maintained and no changes need to be made to the object.
82  - if changes need to be made to the object passed in, pass it by _reference_ (non const).
83  - if a pointer to the object needs to be maintained for later use, pass the the _Ptr_ type as _const T Ptr &_. This maintains a shared pointer to the passed in object. However, this can create a dependency cycle (e.g., StateValidityChecker storing a shared pointer to SpaceInformation, which already stores a shared pointer to StateValidityChecker).
84  - when a dependency cycle would be created, and often access to the pointer is needed, use a _raw pointer_. Use a const raw pointer if possible. If access to the pointer is not often required, using a _weak pointer_ is good.

86 ## Automatic Code Formatting
87 
88 An automatic code formatter is available to automate enforcing the style guide, using [clang-format](http://clang.llvm.org/docs/ClangFormat.html). This set of tools can be used standalone via command line or with editor integrations.
89 
90 ### Setup
91 
92 Install **clang_format**. For Ubuntu:
93 
94  sudo apt-get install -y clang-format-3.6
95 
96 ### Usage
97 
98 A configuration file is available at the base of the OMPL repository that will be automatically picked up by the following commands. You can run `clang_format` in several ways:
99 
100 #### Command Line
101 
102 Format single file:
103 
104  clang-format-3.6 --i -style=file MY_OMPL_FILE.cpp
105 
106 Format entire directory recursively including subfolders:
107 
108  find . -name '*.h' -or -name '*.hpp' -or -name '*.cpp' | xargs clang-format-3.6 --i -style=file $1
109 
110 #### Emacs Editor Configuration
111 
112 In your ``~/.emacs`` config file, add the following:
113 
114 Format your source code if its in some directory such as the ``ompl`` (feel free to change keywords ompl):
115 
116 ```
117 (defun run-ompl-clang-format ()
118  "Runs clang-format on cpp,h files in ompl/ and reverts buffer."
119  (interactive)
120  (and
121  (string-match "/ompl/.*\\.\\(h\\|cpp\\)$" buffer-file-name)
122  (save-some-buffers 'no-confirm)
123  (shell-command (concat "clang-format-3.6 -style=file -i " buffer-file-name))
124  (message (concat "Saved and ran clang-format on " buffer-file-name))
125  (revert-buffer t t t)
126 ))
127 ```
128 
129 Set a keyboard shortcut to run, such as F12
130 
131  (global-set-key [f12] 'run-ompl-clang-format)
132 
133 #### Atom Editor Configuration
134 
135 Install the [clang-format](https://atom.io/packages/clang-format) package via the Atom package manager or ``apm install clang-format``.
136 
137 In the package settings set ``clang-format-3.6`` as your executable and point 'Style' to your ``.clang_format`` file.
